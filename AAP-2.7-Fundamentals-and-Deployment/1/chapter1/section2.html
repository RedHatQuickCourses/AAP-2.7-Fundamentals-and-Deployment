Architecture Overview

How the Platform Works

The following steps describe what happens when an automation job is submitted:

Step 1: User Authentication

  • User logs in through the Platform Gateway (HTTPS port 443)

  • Gateway authenticates against the authentication backend

  • Session token is issued

Step 2: Job Creation

  • User creates a job through the Automation Controller web UI or API

  • Controller validates the job template, inventory, and credentials

  • Job details are stored in the PostgreSQL database

Step 3: Job Execution

  • Controller selects an available execution node (or uses the control plane)

  • Job is dispatched through the Receptor mesh network

  • Execution node pulls the execution environment container image

  • Ansible playbook runs inside the container

Step 4: Results Collection

  • Job output streams back to the controller in real-time

  • Results are stored in the database

  • User can view progress through the web UI

Step 5: Event-Driven Response (if configured)

  • Events from the job can trigger EDA rules

  • EDA can automatically launch additional jobs based on conditions

For example, when deploying a web application, the Controller runs the playbook, execution nodes perform the work on target servers, and if the deployment fails, EDA can automatically trigger a rollback job.

Metrics Service Architecture

AAP 2.7 includes a Metrics Service that collects anonymized usage data. Understanding this helps with capacity planning:

What it collects:

  • Job execution statistics (success/failure counts, duration)

  • Host summary statistics (per host execution performance)

  • Credential type statistics (counts of credential types)

  • Execution environment statistics

  • Collection statistics

  • Ansible Automation Platform configurations like (Project SCM types, Organization counts, version Information, Feature flags, Warnings and Deprecation notices)

What it does NOT collect:

  • Job template names

  • Customer-defined credential names

  • Customer-defined execution environment names

  • Customer-defined collection names

  • Event data

  • Host inventory data

  • Playbook names

  • Variable names and values

  • Workflow template names

Only aggregated statistics and counts are transmitted. No personal data leaves your environment.

Data Flow:

  1. Scheduler triggers collection tasks every hour

  2. Tasks query automation controller database (read-only access)

  3. Raw data is processed and anonymized

  4. Anonymized metrics are stored in the metrics database

  5. Daily at 3:00 AM, data is anonymized and transmitted to Red Hat via Segment

You can disable metrics collection by setting feature flags during installation.
