= Deployment Best Practices

These best practices should be used as a reference point. Because operational requirements vary by organization, you should tailor and design a schedule that aligns with your specific infrastructure and business needs.

== 1. Capacity Planning

Size appropriately based on your environment needs. Consider the following guidelines:

[NOTE]
====
*All-in-one topology — minimum per node:*

*Per virtual machine:*

* RAM: 16 GB minimum
* CPUs: 4 minimum
* Disk: 60 GB total (15 GB installation directory, 1 GB `/var/tmp` for online installs)
* Disk IOPS: 3,000

*System configuration:*

* OS: Red Hat Enterprise Linux 9.4 or later, or RHEL 10 or later
* Database: PostgreSQL 15 (managed automatically)
* Subscriptions: Valid Red Hat AAP subscription and RHEL subscription required

*Prerequisites:*

* Dedicated non-root user with `sudo` privileges on the RHEL host
* SSH public key authentication configured for remote installs
* Required network ports open in the firewall
====

|===
| Environment Type | Managed Nodes | Topology | Resources per Node
| Development | <10 | All-in-one | 8 GB RAM, 2 CPU
| Test/QA | 10-50 | All-in-one | 16 GB RAM, 4 CPU
| Small Production | 50-250 | All-in-one or Small Enterprise | 16-32 GB RAM, 4-8 CPU
| Medium Production | 250-1000 | Enterprise | 32+ GB RAM, 8+ CPU
| Large Production | 1000+ | Enterprise with execution nodes | 64+ GB RAM, 16+ CPU
|===


**Monitor resource usage**: Use the Metrics Service to track job execution patterns and monitor container resource consumption. Regularly review resource utilization trends to ensure the environment can handle current workloads and plan for future growth by adding execution nodes when additional capacity is required.

== 2. High Availability

For production environments requiring HA:

**Components to make highly available**:
1. **Platform Gateway**: Multiple gateway nodes behind a load balancer
2. **Automation Controller**: Multiple controller nodes
3. **Database**: PostgreSQL clustering (primary + replicas)
4. **Automation Hub**: Multiple hub nodes with shared storage
5. **Execution Nodes**: Multiple nodes in a receptor mesh

**Example HA architecture**:

- 2x Platform Gateway nodes (behind HAProxy load balancer)
- 2x Automation Controller nodes
- 1x PostgreSQL primary + 2x replicas
- 2x Automation Hub nodes (with NFS shared storage)
- 3+ Execution nodes

[TIP]
====
For true HA, ensure no single point of failure exists in your design.
====

== 3. Backup and Disaster Recovery

**What to back up**: Ensure that all critical components of the Ansible Automation Platform are included in your backup strategy. This includes PostgreSQL databases for Controller, Automation Hub, and Event-Driven Ansible (EDA), Automation Hub content storage, TLS certificates and keys, inventory files, and any custom configurations required to restore the environment and maintain operational consistency.

**Backup frequency**:

- Databases: Daily minimum, hourly for critical environments
- Configurations: After every change
- Automation Hub content: After publishing new content

**Backup methods**:

**Database backup example**:
```bash
# Taking backup using installer scripts provided by Red Hat
$ ansible-playbook -i <path_to_inventory> ansible.containerized_installer.backup
```

**Database restore example**:
```bash
# Restore hub database installer scripts provided by Red Hat
$ ansible-playbook -i <path_to_inventory> ansible.containerized_installer.restore
```

**Test your backups regularly!**:

- Perform quarterly restore tests
- Document restore procedures
- Verify data integrity after restore

== 4. Maintenance Windows

**Regular maintenance tasks**:

- **Weekly Tasks**: Review failed automation jobs, monitor system disk space usage, and track database size growth to prevent sudden operational disruptions.

- **Monthly Tasks**: Purge old job history via auto-cleanup, update execution environment container images, and review and rotate application logs.

- **Quarterly Tasks**: Apply AAP updates and patches, rotate credentials, audit user access permissions, and test disaster recovery procedures.

- **Annual Tasks**: Plan major platform version upgrades, re-evaluate infrastructure architecture, and conduct a long-term capacity planning review.
