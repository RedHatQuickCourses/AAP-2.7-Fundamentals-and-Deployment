= Security Best Practices

This section covers how to use AAP effectively and securely. These best practices should be used as a reference point. Because operational requirements vary by organization, you should tailor and design a schedule that aligns with your specific infrastructure and business needs.

== 1. Credential Management

Proper credential management is essential for securing your Ansible Automation Platform environment. Protect sensitive information, follow password management best practices, and grant only the minimum permissions required for users and services.

**Do's**:

- ✅ Use Ansible Vault to store sensitive variables in inventory files.
- ✅ Rotate passwords regularly, preferably every 90 days or according to your organization's policy.
- ✅ Use unique passwords for each platform component.
- ✅ Store credentials in AAP Credential Management instead of embedding them in playbooks.
- ✅ Use SSH keys rather than passwords for managed nodes whenever possible.
- ✅ Follow the principle of least privilege and grant only the required level of access.

**Don'ts**:

- ❌ Store passwords in plain-text playbooks or configuration files.
- ❌ Share administrator credentials among team members.
- ❌ Reuse the same password across multiple components.
- ❌ Commit inventory files containing credentials to version control systems.

**Example: Using Ansible Vault**

```bash
# Create a vault file for sensitive variables
ansible-vault create vault.yml
```

Add your passwords:
```yaml
---
gateway_admin_password: "SecurePassword123!"
controller_admin_password: "AnotherSecurePass456!"
hub_admin_password: "YetAnotherPass789!"
```

*Use it during installation*:
```bash
ansible-playbook -i inventory -e @vault.yml --ask-vault-pass \
    ansible.containerized_installer.install
```

== 2. Network Security

A properly configured firewall helps protect the Ansible Automation Platform by limiting access to only the services and users that require it. Review network access regularly and ensure that only necessary communication paths are allowed.

**Firewall Configuration**:

- Only open the ports required for your deployment topology.
- Restrict access to trusted source IP addresses or networks whenever possible.
- Use firewalld or iptables to manage and maintain firewall rules.

**Example firewall configuration**:

```bash
# Allow HTTPS from your corporate network only
sudo firewall-cmd --permanent --add-rich-rule='rule family="ipv4" \
    source address="192.168.0.0/16" port protocol="tcp" port="443" accept'

# Reload firewall
sudo firewall-cmd --reload
```

**TLS/SSL Configuration**:

Secure all production deployments with HTTPS to protect data in transit and prevent unauthorized access. Use certificates issued by a trusted Certificate Authority (CA), disable outdated TLS versions, and enforce modern encryption standards.

- Always use HTTPS in production environments; avoid using HTTP.
- Use certificates from a trusted CA.
- Disable legacy TLS versions and use TLS 1.2 or later, with TLS 1.3 preferred.
- Enable HTTP Strict Transport Security (HSTS) to enforce secure connections.

**Database Security**:

Protect the PostgreSQL databases that support the platform by implementing strong authentication, limiting network access, and securing data both in transit and at rest.

- Use strong passwords for all PostgreSQL accounts.
- Restrict database access to Ansible Automation Platform hosts only.
- Enable SSL for database connections, especially when using remote databases.
- Perform regular encrypted backups to support recovery and data protection.

== 3. Access Control

**Role-Based Access Control (RBAC)**:
- Create teams for different groups (developers, operators, auditors)
- Assign roles with least privilege
  - **System Administrator**: Full access (only for platform admins)
  - **System Auditor**: Read-only access (for compliance/auditing)
  - **Organization Admin**: Full access within their organization
  - **Project Admin**: Manage specific projects only

**Example RBAC structure**:
```
Organization: Production
├── Team: Web Admins
│   └── Role: Admin on "Web Servers" inventory
├── Team: Database Admins
│   └── Role: Admin on "Database Servers" inventory
└── Team: Auditors
    └── Role: Auditor on entire organization
```

== 4. Audit Logging

Enable and monitor audit logs:

**What to monitor**:
- Login attempts (successful and failed)
- Credential usage
- Job launches
- Configuration changes
- User and permission changes

**Integrate with SIEM**:
Send AAP logs to your Security Information and Event Management system for centralized monitoring and alerting on suspicious activity.
